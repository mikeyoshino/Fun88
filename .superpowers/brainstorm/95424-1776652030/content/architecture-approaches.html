Architecture Approaches

Three ways to structure your HTML5 game platform — pick the one that fits best

A

Modular Monolith ★ RECOMMENDED

Single ASP.NET MVC app with feature-based module folders (Games, Admin, Blog, Users, Scraper). Scraper runs as an IHostedService + Quartz.NET inside the same process. Supabase handles auth & file storage; EF Core + PostgreSQL owns all domain data.

Pros

  • One codebase, one Docker image
  • Simple local dev — just dotnet run
  • Modules share DB transaction context
  • Easy to extract later if needed
  • Quartz.NET gives persistent job scheduling

Cons

  • Scraper failures could impact web requests (mitigated by background service isolation)
  • Scaling web and scraper together (not separately)
B

Monolith + Separate Worker Service

ASP.NET MVC web app + a dedicated .NET Worker Service for scraping and translation. Both share the same PostgreSQL database. Docker Compose links both containers. Admin panel triggers the worker via a lightweight internal HTTP call or shared DB job queue table.

Pros

  • Scraper fully isolated — crashes don't touch the web app
  • Can scale scraper container independently
  • Clear process boundary

Cons

  • Two projects to maintain and deploy
  • Local dev needs Docker Compose or two terminals
  • Slightly more complex for this scale
C

Microservices

Separate services: Web API + MVC Frontend, Scraper Service, Translation Service, User Service. API Gateway in front. Each service has its own database schema. Communication via message bus (e.g., RabbitMQ).

Pros

  • Maximum isolation and independent scaling
  • Technology flexibility per service

Cons

  • Massive operational overhead for this use case
  • Distributed tracing, service discovery, network latency
  • Overkill — adds months to delivery with no real benefit at this scale

My Recommendation

Option A — Modular Monolith. At this scale, the scraper runs fine as a background hosted service (Quartz.NET handles persistence and scheduling). You get dotnet run simplicity locally, one Docker image in production, and a clean module structure that could be extracted into Option B later if traffic demands it. The complexity of Option B is only justified when the scraper needs to scale independently from the web tier — unlikely at launch.
